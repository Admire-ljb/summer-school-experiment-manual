实验 3: 初次配置与驱动 Crazyflie 无人机

Initial Crazyflie Configuration and Operation

一、实验目标

在Linux虚拟机上配置crazyfile-client软件。

认识crazyfile无人机的结构。

分别使用有线和无线的方式将虚拟机的客户端与无人机相连接。

使用实例程序试飞一下无人机。

二、实验准备

实验2已经配置好ROS的Ubuntu20.04虚拟机。并且需要具备前几门实验已经积累的linux系统操作经验。

三、实验步骤

实验一：使用 Git 获取 cfclient 指定版本并安装

本节使用 Git 安装 cfclient，并固定到 Bitcraze 官方仓库中兼容 Ubuntu 20.04 默认 Python 3.8 的 2024.7.1 版本。2024.10 及后续版本要求 Python 3.10，不适用于本课程的默认虚拟机环境。打开虚拟机终端，先进入 workspace 并获取源码：

mkdir -p ~/workspace
cd ~/workspace
git clone https://github.com/bitcraze/crazyflie-clients-python.git
cd crazyflie-clients-python
git checkout 2024.7.1

如果已经克隆过该仓库，不需要重新下载；进入已有目录后执行 git fetch --tags，再执行 git checkout 2024.7.1。确认当前终端目录为 ~/workspace/crazyflie-clients-python，后续依赖安装和 cfclient 安装命令均在该目录中执行。

然后，首先我们先安装一些必要的依赖，首先直接复制如下命令粘贴至终端：

sudo apt-get install python3 python3-pip python3-pyqt5 python3-pyqt5.qtsvg

按回车键输入命令，然后可能会要求输入用户密码以获得权限（此过程中已输入的密码内容和数量不可见），以及输入Y继续安装流程。

再输入以下命令安装其他依赖：

sudo apt install libxcb-cursor0 libxcb-xinerama0 libxkbcommon-x11-0

再输入以下命令配置一下pip3使用的软件源：

pip3 config set global.index-url https://pypi.tuna.tsinghua.edu.cn/simple

Ubuntu 20.04 默认使用 Python 3.8，该环境中不应无版本限制地升级打包工具。setuptools 71.x 与旧版 importlib_metadata 同时存在时，后续安装可能出现 AttributeError: module 'importlib_metadata' has no attribute 'EntryPoints'；较新的 pip 和 testresources 也已停止支持 Python 3.8。以下命令固定一组兼容版本。其中，testresources 用于补齐 launchpadlib 的依赖，与 EntryPoints 报错不是同一问题。

python3 -m pip install --user --upgrade "pip==24.3.1" "setuptools==70.3.0" "wheel==0.45.1" "importlib-metadata==8.5.0" "testresources==2.0.1"

故障处理

如果已经执行过无版本限制的升级命令，并在 python3 -m pip install --user -e . 时看到上述 EntryPoints 报错，请重新执行上面的固定版本命令。命令成功后，回到 crazyflie-clients-python 目录继续执行下面的安装命令。

最后输入以下命令进行总的软件依赖安装整理：

python3 -m pip install --user -e .

无报错安装完成之后，重启虚拟机，输入cfclient命令，出现如下界面则代表环境配置安装成功。

具体的无人机基础认知教程可直接打开 Bitcraze 官方教程：

https://www.bitcraze.io/documentation/tutorials/getting-started-with-crazyflie-2-x/

首先来认识一下无人机的基础组成部分，其基础的零件构成如下：

图 1 Crazyflie无人机的零部件

Crazyflie 2.X控制芯片板(即上图零部件中有4个arm的芯片板)在生产过程中已经被充分测试过，但是为了确保运输过程没有导致任何损害，还是需要在组装之前做测试。芯片板如下图，芯片板有四个LED灯(M1, M2, M3, M4)用于显示无人机状态和报错调试等，下图中有彩色的四个点就是，并有对应的标号①②③④。M1灯在右上，有红绿两种灯；M2灯右下，只有蓝灯；M3灯在左下，只有蓝灯；M4灯在左上，有红绿两种灯。

图 2 注意front direction的指向，以及四个LED灯

将Crazyflie 2.X芯片板用USB线连上电源。注意观察LED灯M1和M4， 如果M4会快速闪5次绿灯，则测试通过。

如何读懂LED灯的含义

Crazyflie 2.X如果all good：蓝色LED灯M2和M3(下方的两个灯)会是蓝的(不是闪)，并且M1会闪烁[2次/秒]红灯

Crazyflie 2.X如果all good，但是传感器还没被校准：蓝色LED灯M2和M3(下方的两个灯)会是蓝的(不是闪)，并且M1闪烁[0.5次/秒]红灯，即每2秒闪烁1次.

无线电通讯Radio connected：M4闪烁红and/or绿

电池电量低：M1会是红的(不是闪)

充电中：M3会闪烁蓝灯，同时M2是蓝灯(不是闪)

Bootloader Mode: M2和M3都闪烁蓝灯，约1次/秒

self-test失败：如果M1灯是快速闪5次红灯，并且停顿一段时间，再次重复短闪快速闪5次红灯，再停顿一段时间，如此重复，说明self-test是失败的。注意self-test与组装之前的测试(M4会快速闪5次绿灯，则测试通过。如果M1灯快速闪5次红灯，则失败)是不一样的

实验三：使用有线和无线的方式连接无人机

首先先配置linux虚拟机中的usb设备的权限，详细操作可直接打开 Bitcraze USB permissions 页面：

https://www.bitcraze.io/documentation/repository/crazyflie-lib-python/master/installation/usb_permissions/

具体操作为在原目录的终端中依次输入以下命令：

sudo groupadd plugdev
sudo usermod -a -G plugdev $USER

注意以下命令较长，复制粘贴的时候保证命令完整性：

cat <<EOF | sudo tee /etc/udev/rules.d/99-bitcraze.rules > /dev/null
# Crazyradio (normal operation)
SUBSYSTEM=="usb", ATTRS{idVendor}=="1915", ATTRS{idProduct}=="7777", MODE="0664", GROUP="plugdev"
# Bootloader
SUBSYSTEM=="usb", ATTRS{idVendor}=="1915", ATTRS{idProduct}=="0101", MODE="0664", GROUP="plugdev"
# Crazyflie (over USB)
SUBSYSTEM=="usb", ATTRS{idVendor}=="0483", ATTRS{idProduct}=="5740", MODE="0664", GROUP="plugdev"
EOF

再输入以下命令：

sudo udevadm control --reload-rules
sudo udevadm trigger

进行完以上操作后，输入sudo reboot的命令让虚拟机重启，以应用对usb权限的改动。

然后点击scan按钮，如下显示检测到无人机，选择usb：//0的选项点击connect。

在连接的过程中保证无人机平放在桌面上以使得其能初始化陀螺仪等定位设备，出现以下界面则代表成功以有线的方式连接，此时手拿起无人机可以看到姿态界面中发生的变化。

接下来，我们查看一下无人机上的无线地址，记住这个地址，选中复制，粘贴到主界面的address栏中，我们将会用crazyradio来以无线的方式连接到无人机。

将无人机的usb线取下，插上crazyradio，用cfclient客户端scan到该设备，注意保证下方的address中的地址为刚才查看到的无人机的无线连接地址，再点击connect

同样在连接的过程中保证无人机静止平放，使其能初始化姿态，出现如下界面，手上拿着无人机，姿态界面出现相应的变化，则以无线的方式连接成功。

多组同时实验的无线频段规划

正式实验中如果 12 组在同一房间同时使用 Crazyradio，建议为每组固定一套完整的 radio URI。URI 格式为 radio://接口编号/信道/速率/地址，例如 radio://0/60/2M/E7E7E7E7A6。其中，信道用于减少同频无线拥塞，地址用于避免误连到其他组的无人机；只修改地址不能完全解决同频干扰问题。

配置时先用 USB 单独连接一架无人机，在 cfclient 中连接 usb://0，进入 Connect - Configure 2.x，写入该组分配的 Radio channel 和 Radio address，保存后重启无人机。这个操作只修改无人机的无线配置，不会自动同步到 Python 代码或虚拟机。

将完整 URI 保存到虚拟机

每组在自己的虚拟机中执行一次下面的命令。终端出现提示后，从上表复制并粘贴本组的完整 URI，不要只输入 address：

read -r -p "Paste group URI: " CFLIB_URI
export CFLIB_URI
printf "export CFLIB_URI='%s'\n" "$CFLIB_URI" >> ~/.bashrc
echo "$CFLIB_URI"

该命令将完整 URI 保存为 CFLIB_URI 环境变量，以后打开新终端也会自动生效。后续实验中的代码统一通过 uri_helper.uri_from_env() 读取该变量，无需在每个脚本中重复修改地址。如果组别 URI 发生变化，请在 ~/.bashrc 中更新 export CFLIB_URI=... 这一行。

也可以仅对某一次运行临时指定 URI。下面以 6 组运行 test.py 为例，使用时必须替换为本组 URI 和实际脚本名：

CFLIB_URI='radio://0/60/2M/E7E7E7E7A6' python3 test.py

课堂注意事项

同一房间多组实验时，信道尽量拉开，例如间隔 10；每架无人机和每台电脑旁应贴上组别、channel、address 和完整 URI。正式飞行前先逐组单独测试连接，确认不会扫描或连接到其他组无人机。

实验四：使用python脚本让无人机自主起飞

核心部件

Flow Deck： Flow Deck 是一款激光雷达测距传感器板，用于测量Crazyflie无人机的高度和距离地面的距离。它可以帮助无人机进行高度定位和稳定悬停。

确保前面的操作已经可以使用crazyradio以无线的方式连接到无人机，

使用 Crazyradio 扫描和连接前，确认 cfclient 地址栏中的接口、channel、速率和 address 与本组完整 URI 一致；不能只核对 //0/80/2M 这一部分。

将 test.py 放入 workspace 文件夹。脚本通过 uri_helper.uri_from_env() 读取已经保存的 CFLIB_URI，不再在代码中手工填写小组地址。运行前先执行 echo "$CFLIB_URI"，确认输出与本组完整 URI 一致。

再安装最后一个所需的依赖，在终端中输入如下命令：

pip3 install cflib

然后在终端中输入python3 test.py命令运行该脚本，脚本开始运行后，先等待一段时间的初始化，通过无线的方式向无人机发送指令，无人机接收到指令后进行相印的飞行动作。

补充，实验官方教程网址为：

https://www.bitcraze.io/documentation/tutorials/getting-started-with-stem-drone-bundle/

test.py脚本内容为：

下载经审核的完整脚本: motion_commander_sequence.py

#!/usr/bin/env python3
"""Finite MotionCommander sequence with Flow-deck and URI checks."""

import threading
import time

import cflib.crtp
from cflib.crazyflie import Crazyflie
from cflib.crazyflie.syncCrazyflie import SyncCrazyflie
from cflib.positioning.motion_commander import MotionCommander
from cflib.utils import uri_helper


URI = uri_helper.uri_from_env(default="")
flow_deck_attached = threading.Event()


def request_arming(scf, armed):
    for service_name in ("platform", "supervisor"):
        service = getattr(scf.cf, service_name, None)
        request = getattr(service, "send_arming_request", None)
        if request is not None:
            request(armed)
            return
    raise RuntimeError("This cflib version has no arming service.")


def flow_deck_callback(name, value):
    if int(value):
        flow_deck_attached.set()


def main():
    if not URI:
        raise RuntimeError("CFLIB_URI is empty; configure the group URI first.")
    cflib.crtp.init_drivers(enable_debug_driver=False)
    with SyncCrazyflie(URI, cf=Crazyflie(rw_cache="./cache")) as scf:
        scf.cf.param.add_update_callback(
            group="deck", name="bcFlow2", cb=flow_deck_callback
        )
        time.sleep(1.0)
        if not flow_deck_attached.wait(timeout=5.0):
            raise RuntimeError("Flow deck was not detected; takeoff cancelled.")

        armed = False
        try:
            request_arming(scf, True)
            armed = True
            time.sleep(1.0)
            with MotionCommander(scf, default_height=0.30) as commander:
                time.sleep(1.0)
                commander.forward(0.50, velocity=0.20)
                time.sleep(1.0)
                commander.up(0.20, velocity=0.15)
                time.sleep(1.0)
                commander.circle_right(0.50, velocity=0.20, angle_degrees=270)
                commander.down(0.20, velocity=0.15)
                time.sleep(1.0)
                commander.left(0.20, velocity=0.20)
                time.sleep(1.0)
                commander.forward(0.50, velocity=0.20)
        finally:
            if armed:
                request_arming(scf, False)


if __name__ == "__main__":
    main()
